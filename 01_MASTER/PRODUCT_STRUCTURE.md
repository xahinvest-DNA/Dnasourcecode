# Product Structure

## 1. Product Purpose At System Level
The product is a system for identifying one repeating money/income mechanism, making its cycle visible, running one bounded restructuring cycle against it, and preserving the result as reusable memory for later cycles.

It exists to prevent the product from collapsing into loose chat, isolated insight, or motivational output.

## 2. Product Thesis
The user does not mainly suffer from lack of information.
The user is held inside a repeating hidden poverty mechanism that reproduces the same result through a self-reinforcing cycle.

The product creates value when it:
- identifies the mechanism,
- maps the current cycle,
- defines one admissible replacement cycle,
- converts that replacement into one bounded action,
- records what happened after the action.

## 3. Product Levels / Layers

### Strategic Structure Layer
Owns the meaning of the product as a system.
Defines what a mechanism is, what a cycle is, what value is, what memory is for, and what each intelligence layer owns.

### Product Flow Layer
Owns the canonical MVP money/income flow, screen sequence, artifact chain, and UX rules.

### Module Contract Layer
Owns module responsibilities, inputs, outputs, validation boundaries, and interpretation boundaries.

### Technical Model Layer
Owns domain model, data schema, state model, orchestration rules, and implementation-facing structures.

## 4. Strategic Entities
- `User`: the person moving through one money/income restructuring cycle at a time.
- `Hidden Poverty Mechanism`: the leading mechanism hypothesis that explains the current repeated result.
- `Cycle`: the linked sequence from belief through reinforcement.
- `Old Cycle`: the current repeating loop that reproduces the undesired money/income outcome.
- `Replacement Cycle`: the admissible new loop the product proposes for testing.
- `Micro-Action`: the one bounded real-world test for the replacement cycle.
- `Check-In Result`: the recorded outcome after the user attempts the action.
- `Progress Memory`: the stored cycle result and repeating historical markers.

## 5. Definition Of Hidden Poverty Mechanism
A hidden poverty mechanism is the leading internal pattern that keeps reproducing a constrained money/income result.

It is not just a sentence or opinion.
It is the repeatable internal logic that:
- shapes what the user treats as true,
- filters what the user notices,
- influences what the user repeatedly does or avoids,
- helps recreate the same result,
- makes that result feel self-confirming.

In MVP, the product works with one leading mechanism hypothesis per cycle.

## 6. Definition Of Cycle
A cycle is the minimum strategic unit of causation in the product:
`belief -> attention -> behavior -> result -> reinforcement`

This definition is product-level truth, not only UX language.

The cycle matters because the product is not designed to produce abstract insight alone.
It is designed to intervene in one repeating loop and test whether a different loop can begin.

## 7. Definition Of Completed Restructuring Cycle
A completed restructuring cycle is the minimum value unit of the product.

It is complete only when all of the following exist:
- one `Diagnosis Output`,
- one `Old Cycle Map`,
- one `Restructuring Output`,
- one `Action Output`,
- one `Check-In Output`,
- one `Progress Snapshot`.

Insight without action is not a completed cycle.
Action without check-in is not a completed cycle.

## 8. Cycle Status Types

### `diagnosed`
The old mechanism and old cycle are visible, but restructuring has not yet been finalized.

### `restructured`
The replacement cycle exists, but the action has not yet been assigned or attempted.

### `action_assigned`
One bounded action exists and is awaiting attempt.

### `checkin_received`
The user has returned evidence from the action attempt.

### `completed_shifted`
The cycle closed with a visible shift marker that supports the replacement cycle.

### `completed_partial`
The cycle closed with partial movement but without a strong shift marker.

### `completed_blocked`
The cycle closed with no effective shift and the old cycle remained dominant.

These are strategic status types for product meaning.
They are not yet a runtime state machine.

## 9. Memory Boundaries
Memory stores the result of completed or attempted cycles so later cycles do not start from zero.

Memory does store:
- prior mechanism hypotheses,
- prior old and replacement cycle outcomes,
- action completion history,
- repeated barriers,
- shift markers.

Memory does not store:
- the right to redefine the product flow,
- the authority to open the next manager packet,
- broad personality claims outside the money/income scenario,
- open-ended narrative history that does not help the next cycle.

## 10. Product Expansion Logic
The MVP stays inside money/income only.

Expansion does not happen by broadening the current flow first.
Expansion happens only after the money/income cycle is stable as a reusable product structure.

Allowed future expansion direction:
1. prove the restructuring system in money/income,
2. stabilize cycle memory and action quality,
3. only then consider other domains that share the same system logic.

## 11. Decision Boundary

### Deterministic Code
Owns:
- flow order,
- state discipline,
- schema discipline,
- persistence,
- packet boundaries,
- guardrails.

Does not own:
- hidden-structure interpretation,
- adaptive reframing language.

### LLM
Owns:
- adaptive reframing language,
- conversion of accepted artifacts into user-ready language,
- generation of the replacement formulation and one action within fixed product boundaries,
- interpretation of check-in language inside the defined cycle.

Does not own:
- product flow definition,
- packet authority,
- independent product strategy.

### Communication DNA
Owns:
- hidden-structure interpretation support,
- detection of prohibition, resistance, and self-sabotage cues from user language.

Does not own:
- final diagnosis artifact,
- flow routing,
- action assignment,
- whole-product logic.

## 12. Strategic Constraints
- The system is a trainer, not an affirmation app.
- The MVP is money/income only.
- The product owns one canonical restructuring flow.
- The product owns one leading mechanism hypothesis per cycle.
- The product owns one bounded action per cycle.
- The product measures value in completed restructuring cycles, not in chat volume or insight count.
- The strategic layer must not drift into runtime schema or implementation design.
