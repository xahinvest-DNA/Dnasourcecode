# Roadmap

| Phase | Title | Goal | Exit Criteria |
| --- | --- | --- | --- |
| F-001 | Repository / Project Brain Initialization | Create the repository operating system and place accepted context into it. | Master, product, module, tech, and Codex control docs exist with initial real content; active packet is documented; worklog is initialized. |
| F-002 | Product Framing | Lock user-facing framing, flow order, screen map, and MVP structure. | Product docs reflect a stable first-cycle flow and manager accepts the framing. |
| F-003 | Module Boundary Fixation | Define clear responsibilities, inputs, outputs, and non-goals of each module. | Module contracts are stable enough to guide implementation packets. |
| F-003a | Product Structure Fixation | Add the missing strategic structure layer between accepted product/module truth and the upcoming tech model. | Product structure is explicit enough that the tech layer does not need to reinterpret mechanism, cycle, memory, or intelligence ownership. |
| F-004 | Tech Model Fixation | Fix the domain model, state model, data schema, orchestration rules, and structured outputs. | Technical truth is implementation-ready for the first executable slice. |
| F-005 | First MVP Flow Packet | Build one very narrow working vertical slice of the MVP. | One end-to-end executable cycle exists for intake -> diagnosis -> restructuring -> action -> check-in history. |

## Sequencing Rule
- Do not skip phases.
- Do not open broad runtime implementation before `F-004` is accepted.
- Keep each next packet bounded, reviewable, and visible to the user or manager.
