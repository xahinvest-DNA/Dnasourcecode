# UX Principles

## Framing Rules
- Start from a live money/income pain point, not from abstract self-development language.
- Frame the product as a trainer for a repeating mechanism, not as coaching, therapy, or motivation.
- Explain the current loop before presenting any reframing.

## Interaction Rules
- Ask only for information needed for the current stage.
- Keep every screen focused on one decision or one artifact.
- Use short operational language; avoid inspirational copy and abstract promises.
- Present one leading interpretation, not a menu of possibilities.
- Keep the user in a forward flow; avoid exploratory detours in MVP.

## Output Rules
- Diagnosis must name the current mechanism in a way the user can recognize.
- Restructuring must feel admissible, not maximally positive.
- Action must be concrete enough to complete or fail clearly.
- Check-in must ask for evidence of what happened, not only for feelings.
- Progress must show cycle status, not a generic encouragement message.

## Constraint Rules
- Never assign more than one real-world action in one cycle.
- Never collapse diagnosis, restructuring, and action into one screen output.
- Never introduce non-money examples in the MVP flow.
- Never use the product layer to imply runtime implementation choices.

## Quality Bar
- If the user can leave a screen without knowing what was produced there, the screen is too vague.
- If the user receives an output that cannot be passed cleanly to the next stage, the flow is too loose for MVP.
